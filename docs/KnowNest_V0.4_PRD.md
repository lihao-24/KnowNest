# KnowNest V0.4 需求文档

版本：V0.4  
主题：AI 类型模板化整理  
状态：需求设计稿  
适用项目：KnowNest 个人知识库 Web App  
生成日期：2026-04-30

---

## 1. 版本背景

KnowNest 当前已完成 V0.1、V0.2、V0.3 三个阶段。

V0.3 已上线 AI 知识整理助手，支持 AI 生成摘要、推荐标签、推荐分类、优化标题、整理正文，并且 AI 只返回候选结果，由用户预览并确认后才写入知识数据。

V0.4 不再继续扩展 AI 的数量型功能，而是围绕一个核心问题进行增强：

> 用户可以自由输入内容，AI 根据知识类型自动整理成结构化模板格式。

V0.4 的重点是让 AI 从“帮我优化内容”升级为“帮我把杂乱内容整理成适合长期保存的知识结构”。

---

## 2. 版本定位

```text
KnowNest V0.4：AI 类型模板化整理
```

核心目标：

```text
类型决定模板结构
分类提供领域上下文
标签提供关键词补充
AI 负责按模板整理
用户预览后再应用
```

用户不需要手动按照模板填写内容，而是可以先随意记录，再让 AI 根据当前知识类型匹配模板，自动整理成结构化 Markdown 内容。

## 3. 目标用户场景

### 3.1 随手记录项目经验

用户输入：

```text
今天修了一个筛选 bug，列表页切换分类后没有重新请求数据，后来发现 useEffect 依赖漏了 categoryId。
```

选择类型：

```text
项目记录
```

AI 整理后：

```md
## 背景

今天处理列表页筛选相关问题。

## 问题

切换分类后，列表页没有重新请求数据。

## 原因

`useEffect` 依赖数组中遗漏了 `categoryId`，导致分类变化后副作用没有重新触发。

## 解决方案

将 `categoryId` 补充到 `useEffect` 依赖数组中。

## 经验总结

遇到筛选条件变化但页面数据未刷新时，应优先检查状态更新、请求触发逻辑和 `useEffect` 依赖数组。
```

### 3.2 整理 Prompt

用户输入一段临时 Prompt，选择类型：

```text
Prompt
```

AI 自动整理为：

```md
## 使用场景

## Prompt 正文

## 输入变量

## 输出要求

## 示例

## 优化记录
```

### 3.3 整理链接收藏

用户复制一个链接和几句备注，选择类型：

```text
链接
```

AI 自动整理为：

```md
## 链接

## 内容摘要

## 重点信息

## 为什么收藏

## 适用场景

## 相关标签
```

## 5. 核心概念设计

### 5.1 类型

类型是知识内容的结构主维度。

当前类型包括：

```text
笔记
链接
Prompt
项目记录
日志
摘录
计划
代码片段
```

在 V0.4 中，类型用于决定 AI 整理时采用哪种正文结构。

### 5.2 分类

分类是知识内容的领域维度。

例如：

```text
学习
工作
生活
项目
灵感
其他
```

分类不直接决定模板，但会作为 AI 整理上下文。

例如：

```text
类型：项目记录
分类：工作
```

AI 应更偏向项目执行、问题解决、行动项整理。

### 5.3 标签

标签是知识内容的关键词维度。

例如：

```text
#React
#AI
#Prompt
#bug修复
#项目经验
```

标签不作为 V0.4 的主要模板绑定依据，但可以辅助 AI 理解内容重点。

### 5.4 模板

模板用于定义某一类知识的 Markdown 结构和 AI 整理要求。

模板由用户管理，可以新增、编辑、删除、启用或停用。

## 6. 模板管理模块

### 6.1 页面入口

建议页面路径：

```text
/app/settings/templates
```

侧边栏或设置页中新增入口：

```text
设置
  - AI 设置
  - 知识模板
```

如果当前设置页结构较简单，也可以先在设置页内增加一个“知识模板”区域，不强制拆分新页面。

### 6.2 模板字段

每个模板包含以下字段：

```text
模板名称
适用类型
模板描述
模板正文结构
AI 整理要求
是否启用
是否为该类型默认模板
创建时间
更新时间
```

建议字段说明：

| 字段 | 说明 | 是否必填 |
|---|---|---|
| name | 模板名称 | 是 |
| type | 适用知识类型 | 是 |
| description | 模板用途说明 | 否 |
| content | Markdown 模板结构 | 是 |
| aiInstructions | AI 整理要求 | 否 |
| isEnabled | 是否启用 | 是 |
| isDefault | 是否为该类型默认模板 | 是 |

### 6.3 模板列表页

模板列表应展示：

```text
模板名称
适用类型
是否默认
是否启用
更新时间
操作按钮
```

操作包括：

```text

复制
启用 / 停用
设为默认
删除
```

V0.4 最小可行版本可以先只做：

```text
新增
编辑
删除
设为默认
```

### 6.4 模板编辑页 / 弹窗

模板编辑区域包括：

```text

模板正文 Markdown 编辑区
AI 整理要求输入区
是否启用开关
是否设为该类型默认模板
保存按钮
```

模板正文结构示例：

```md
## 背景

## 问题

## 原因

## 解决方案

## 经验总结

## 后续行动
```

AI 整理要求示例：

```text
请严格根据用户原始内容整理，不要编造不存在的信息。
如果某个部分没有相关内容，可以写“未提及”。
优先保留具体问题、解决过程、决策和行动项。
```

## 7. 类型绑定模板规则

### 7.1 基础规则

V0.4 采用简单稳定的匹配规则：

```text
用户手动选择模板
>
当前类型默认模板
>
全局默认模板
```

不做复杂规则引擎。

### 7.2 类型默认模板

每个类型最多只能有一个默认模板。

类型值必须使用项目受控常量，前端和 AI 都不能提交任意字符串作为知识类型。当前受控类型由代码层维护：

```text
note
link
prompt
project
log
excerpt
plan
snippet
```

例如：

```text
类型：Prompt
默认模板：Prompt 模板

类型：项目记录
默认模板：项目记录模板

类型：代码片段
默认模板：代码片段模板
```

当用户在知识编辑页选择类型后，系统自动匹配该类型的默认模板。

默认模板唯一性必须同时由业务逻辑和数据库部分唯一索引保证，不能只依赖前端状态或 Server Action 顺序。

### 7.3 全局默认模板

全局默认模板用于兜底，不绑定具体知识类型。

数据表达建议：

```text
type = null
is_global_default = true
```

每个用户最多只能有一个全局默认模板。

V0.4 最小版本可以不提供“设置全局默认模板”的显式 UI，但数据模型和匹配逻辑需要支持该回退层：

```text
用户手动选择模板
>
当前类型默认模板
>
全局默认模板
>
无可用模板提示
```

### 7.4 手动切换模板

在知识编辑页中，用户可以看到当前匹配模板：

```text
当前模板：项目记录模板    [切换模板]
```

点击“切换模板”后，可以选择其他启用中的模板。

手动选择的模板只影响当前这次 AI 整理，不一定修改该类型的默认模板。

## 8. 内置默认模板

V0.4 初始内置 8 个模板，分别对应当前 8 个知识类型。

### 8.1 笔记模板

适用类型：

```text
笔记
```

模板内容：

```md
## 核心内容

## 关键要点

## 我的理解

## 相关知识

## 后续行动
```

AI 整理要求：

```text
将原始内容整理为清晰的学习或思考笔记。
保留用户自己的观点和理解。
不要扩展无依据的内容。
```

### 8.2 链接模板

适用类型：

```text
链接
```

模板内容：

```md
## 链接

## 内容摘要

## 重点信息

## 为什么收藏

## 适用场景

## 相关标签
```

AI 整理要求：

```text
识别正文中的链接和用户备注。
如果用户没有提供链接内容详情，不要假装已经阅读链接。
可以根据用户提供的备注整理收藏原因和适用场景。
```

### 8.3 Prompt 模板

适用类型：

```text
Prompt
```

模板内容：

```md
## 使用场景

## Prompt 正文

## 输入变量

## 输出要求

## 示例

## 优化记录
```

AI 整理要求：

```text
将 Prompt 整理为可复用格式。
保留原始 Prompt 的核心意图。
如存在变量，提取为输入变量。
如没有示例，可写“未提供”。
```

### 8.4 项目记录模板

适用类型：

```text
项目记录
```

模板内容：

```md
## 背景

## 目标

## 当前进展

## 问题与原因

## 解决方案

## 决策记录

## 后续行动
```

AI 整理要求：

```text
用于整理项目经验、开发记录、问题修复和阶段复盘。
优先提取背景、问题、原因、解决方案和下一步行动。
不要编造不存在的项目结论。
```

### 8.5 日志模板

适用类型：

```text
日志
```

模板内容：

```md
## 日期

## 记录内容

## 重要事件

## 感受与观察

## 总结

## 下一步计划
```

AI 整理要求：

```text
将用户的日常记录整理为清晰日志。
保留用户的感受和观察。
如果没有日期，可以写“未提及”。
```

### 8.6 摘录模板

适用类型：

```text
摘录
```

模板内容：

```md
## 原文摘录

## 出处

## 核心观点

## 我的理解

## 可复用场景

## 相关知识
```

AI 整理要求：

```text
区分原文摘录和用户自己的理解。
不要擅自改写原文摘录的含义。
如果出处未提供，写“未提及”。
```

### 8.7 计划模板

适用类型：

```text
计划
```

模板内容：

```md
## 目标

## 背景

## 任务清单

## 时间安排

## 风险与阻碍

## 下一步行动
```

AI 整理要求：

```text
将用户的想法整理为可执行计划。
优先提取目标、任务、时间安排和下一步行动。
不要添加用户没有提到的硬性截止时间。
```

### 8.8 代码片段模板

适用类型：

```text
代码片段
```

模板内容：

```md
## 使用场景

## 代码

## 参数说明

## 使用方式

## 注意事项

## 相关链接
```

AI 整理要求：

```text
保留代码原文，不要随意改写代码逻辑。
可以解释代码用途、参数和注意事项。
如果代码不完整，应明确说明。
```

## 9. AI 按模板整理功能

### 9.1 功能入口

在知识新建页和编辑页中增加按钮：

```text
AI 按模板整理
```

按钮位置建议放在正文编辑区域附近，和已有 AI 助手入口保持一致。

### 9.2 触发条件

用户点击后，系统需要检查：

```text
1. 当前用户已登录
2. 当前知识属于当前用户
3. 正文内容长度满足 AI_MIN_INPUT_CHARS
4. 正文内容不超过 AI_MAX_INPUT_CHARS
5. 当前类型存在可用模板
6. 当前模板正文不为空且格式合法
7. 当前模板未被删除或停用
8. 当前用户未超过每日 AI 使用限制
```

如果不满足条件，展示明确错误提示。

### 9.3 整理弹窗

点击后展示整理确认弹窗：

```text
AI 模板化整理

当前类型：项目记录
使用模板：项目记录模板

[开始整理]
```

V0.4 最小版本默认整理标题、摘要、正文、标签和分类，不做勾选整理范围的 UI。

### 9.4 AI 输入内容

调用 AI 时应包含：

```text
知识标题
知识正文
当前类型
当前分类
当前标签
当前模板正文结构
当前模板 H2 段落清单
当前模板 AI 整理要求
当前用户选择的模型
```

模板 H2 段落清单由系统从模板正文中解析 `## ` 标题得到，而不是交给 AI 自行推断。

AI 输出正文必须满足：

```text
1. H2 段落与模板 H2 段落清单完全一致。
2. H2 段落顺序与模板一致。
3. 模板中存在但原文没有信息的段落，填写“未提及”。
4. 不额外新增模板外的 H2 段落。
5. 用户自定义模板同样适用以上规则。
```

### 9.5 AI 输出内容

AI 应返回结构化候选结果：

```ts
{
  title: string;
  summary: string;
  content: string;
  tags: string[];
  category: string | null;
  reasoning?: string;
}
```

其中：

```text
title：优化后的标题
summary：摘要
content：按模板整理后的 Markdown 正文
tags：推荐标签
category：推荐分类
reasoning：可选，简单说明整理依据，不展示复杂推理
```

### 9.6 AI 输出校验规则

AI 输出必须经过运行时 schema 校验后才能进入预览 UI。可以使用 Zod、JSON Schema，或项目现有的显式校验函数；不能只依赖 TypeScript 类型。

校验规则：

```text
1. Provider 必须以 JSON 模式或等价的结构化输出方式返回结果。
2. JSON 解析失败时，进入“AI 返回格式异常”错误分支。
3. title 必须是非空字符串。
4. summary 必须是非空字符串。
5. content 必须是非空字符串，并通过模板 H2 段落一致性校验。
6. content 长度不能超过服务端配置的 AI 输出长度上限。
7. tags 必须是字符串数组，去空、去重后最多保留 8 个。
8. category 只能是当前用户已有分类名称或 null。
9. reasoning 可选，最长保留 200 字，仅用于前端临时展示。
10. reasoning 不入库，不写入 AI 使用日志。
```

### 9.7 重要限制

AI 不允许直接写入知识数据。

AI 只返回候选结果，用户必须手动确认应用。

这延续 V0.3 已有边界：

```text
/api/ai 只生成候选结果，不直接覆盖知识数据。
所有应用动作都必须校验当前用户和知识归属。
AI 日志只保存元信息，不保存完整正文。
```

## 10. AI 整理前后对比

### 10.1 对比视图

AI 返回结果后，展示前后对比：

桌面端：

```text
左侧：原始正文
右侧：AI 整理后正文
```

移动端：

```text
上方：原始正文
下方：AI 整理后正文
```

V0.4 不要求做复杂 diff 高亮，只要求用户可以同时查看原文和 AI 结果。

### 10.2 对比内容

对比区域至少包含：

```text
原始标题 vs AI 标题
原始正文 vs AI 正文
原始标签 vs 推荐标签
原始分类 vs 推荐分类
```

如果空间有限，可以优先对比正文，标题、标签、分类以卡片形式展示。

## 11. AI 结果分区应用

### 11.1 应用方式

AI 结果展示后，用户可以选择：

```text
应用标题
应用摘要
应用正文
应用标签
应用分类
全部应用
取消
```

### 11.2 应用规则

#### 应用标题

将 AI 推荐标题覆盖当前标题。

#### 应用摘要

将 AI 生成摘要写入摘要字段。

#### 应用正文

将 AI 模板化正文覆盖当前正文。

#### 应用标签

推荐标签默认追加到当前标签中。

如果标签已存在，不重复添加。

标签合并后必须遵守项目现有标签数量上限。如果当前项目尚未定义显式上限，V0.4 前端最多展示并应用 20 个标签，超过部分不自动应用，并提示用户手动选择。

#### 应用分类

如果 AI 推荐分类已存在，则选中对应分类。

如果分类不存在，V0.4 不建议自动创建分类，可以提示：

```text
AI 推荐分类「xxx」当前不存在，请手动选择或先创建分类。
```

可以提供次级操作“创建并选中该分类”，但该操作必须走现有分类创建权限校验，不允许 AI 自动静默创建分类。

## 12. 数据库设计建议

### 12.1 新增表：knowledge_templates

建议新增模板表：

```sql
CREATE TABLE knowledge_templates (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid NOT NULL,
  name text NOT NULL,
  type text,
  description text,
  content text NOT NULL,
  ai_instructions text,
  is_enabled boolean NOT NULL DEFAULT true,
  is_default boolean NOT NULL DEFAULT false,
  is_global_default boolean NOT NULL DEFAULT false,
  source text NOT NULL DEFAULT 'user',
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);
```

字段说明：

```text
type：受控知识类型；全局默认模板使用 null。
is_default：是否为某个类型的默认模板。
is_global_default：是否为用户的全局默认模板。
source：模板来源，取值 builtin / user。内置默认模板初始化到用户账号下时标记为 builtin，用户新增模板标记为 user。
updated_at：需要通过数据库 trigger 或 Server Action 统一更新。
```

约束建议：

```sql
CREATE UNIQUE INDEX uniq_default_template_per_type
  ON knowledge_templates (user_id, type)
  WHERE is_default = true AND type IS NOT NULL;

CREATE UNIQUE INDEX uniq_global_default_template
  ON knowledge_templates (user_id)
  WHERE is_global_default = true;
```

类型字段必须由代码层 `KNOWLEDGE_TYPES` 常量校验，禁止前端或 AI 提交任意字符串。

### 12.2 默认模板归属

有两种方案。

#### 方案 A：默认模板写入每个用户

用户首次进入模板管理页时，系统为该用户创建 8 个默认模板。

如果用户绕过模板管理页，首次在新建页或编辑页触发模板化整理前，系统也应检查并初始化默认模板。

初始化规则：

```text
1. 当前用户没有任何模板时，插入 8 个内置默认模板。
2. 当前用户已有模板时，不覆盖用户数据。
3. 初始化逻辑集中在 repository 层，页面只调用同一个 ensureDefaultKnowledgeTemplates(userId)。
4. 不在多个页面重复手写 upsert 逻辑。
```

优点：

```text
用户可以自由编辑默认模板
权限模型简单
查询简单
```

缺点：

```text
每个用户都会有一份默认模板数据
```

个人项目下推荐方案 A。

#### 方案 B：系统模板 + 用户模板

模板区分：

```text
system
user
```

优点：

```text
系统默认模板统一管理
```

缺点：

```text
权限和覆盖逻辑更复杂
```

V0.4 不建议采用方案 B。

### 12.3 默认模板唯一性

同一个用户、同一个类型，只允许一个默认模板。

可以在业务逻辑中保证：

```text
当某个模板设为默认时，将该用户同类型的其他模板 is_default 置为 false。
```

数据库必须增加部分唯一索引兜底，避免并发请求导致同一类型出现多个默认模板：

```sql
CREATE UNIQUE INDEX uniq_default_template_per_type
  ON knowledge_templates (user_id, type)
  WHERE is_default = true AND type IS NOT NULL;
```

全局默认模板同样必须唯一：

```sql
CREATE UNIQUE INDEX uniq_global_default_template
  ON knowledge_templates (user_id)
  WHERE is_global_default = true;
```

### 13.2 AI 模板化整理

可以在现有 `/api/ai` 基础上扩展 action 类型：

```text
template_organize
```

该 action 应接入 V0.3 已有的统一 AI dispatch / schema 校验层，复用认证、知识归属、限流、Provider 选择、日志和错误处理，不为模板整理单独复制一套鉴权与限流逻辑。

请求示例：

```ts
{
  action: "template_organize",
  itemId?: string;
  title: string;
  content: string;
  type: string;
  category?: string;
  tags: string[];
  templateId: string;
  modelId?: string;
}
```

返回示例：

```ts
{
  title: string;
  summary: string;
  content: string;
  tags: string[];
  category: string | null;
}
```

服务端处理要求：

```text
1. 先解析并校验 action 请求 schema。
2. 再校验用户、知识归属和模板归属。
3. 再校验输入长度、模板有效性和每日限流。
4. Provider 返回后执行 AI 输出 schema 校验。
5. 校验失败时只记录 error 元信息，不进入预览，不写入知识数据。
```

## 15. 错误处理

需要覆盖以下错误：

```text
输入内容太短
输入内容太长
没有可用模板
模板正文为空或格式不合法
模板被删除或停用
AI 使用次数已达上限
模型未配置
模型调用失败
AI 返回格式异常
当前知识不存在
当前知识不属于当前用户
网络错误
```

错误提示示例：

```text
当前类型暂无可用模板，请先在设置中创建模板。
```

```text
AI 整理失败：当前模型暂时不可用。你可以稍后重试，或在设置中切换模型。
```

```text
内容超过当前 AI 输入长度限制，请缩短正文后再试。
```

如果用户手动选择的模板在整理前已被删除或停用，系统应自动回退到当前类型默认模板；如果当前类型默认模板也不可用，再回退到全局默认模板；仍不可用时展示“当前类型暂无可用模板，请先在设置中创建模板。”

## 16. 权限与安全要求

V0.4 必须继续遵守现有安全边界：

```text
1. 前端不得直接调用模型 Provider。
2. 不得使用 NEXT_PUBLIC_* 环境变量保存 AI Provider 密钥。
3. AI Provider API Key 只能存在服务端环境变量中。
4. AI 只返回候选结果，不直接覆盖知识数据。
5. 用户只能访问自己的模板。
6. 用户只能将 AI 结果应用到自己的知识条目。
7. AI 日志只保存元信息，不保存完整正文。
8. 模板 aiInstructions 属于不可信用户输入，必须通过 system prompt 边界隔离，不能作为系统指令覆盖安全规则。
9. AI 输出 reasoning 仅前端临时展示，不入库，不写入 AI 使用日志。
10. 非法 JSON、schema 校验失败、模板结构不一致时，不得写入知识数据，只能记录错误元信息。
```

## 17. 验收标准

### 17.1 模板管理

验收条件：

```text
用户可以进入知识模板管理页面。
用户可以看到 8 个默认类型模板。
用户可以新建模板。
用户可以编辑模板名称、类型、正文结构和 AI 整理要求。
用户可以删除自己创建的模板。
用户可以将某个模板设为对应类型的默认模板。
同一类型同时只能有一个默认模板。
数据库唯一索引能阻止并发设默认导致多个默认模板。
每个用户最多只能有一个全局默认模板。
```

### 17.2 类型模板匹配

验收条件：

```text
用户在编辑知识时选择类型，系统能自动匹配该类型默认模板。
用户可以在编辑页看到当前匹配模板。
用户可以手动切换其他模板。
如果没有模板，系统展示明确提示。
手动选择模板被删除或停用时，系统能按规则回退到类型默认或全局默认模板。
```

### 17.3 AI 按模板整理

验收条件：

```text
用户输入自由文本后，可以点击 AI 按模板整理。
AI 能根据当前模板输出结构化 Markdown 正文。
AI 输出正文的 H2 段落与模板 H2 段落完全一致且顺序一致。
缺失信息的模板段落显示“未提及”。
AI 输出包含标题、摘要、正文、标签和分类候选结果。
AI 返回非法 JSON 或 schema 校验失败时，原知识内容不变。
AI 不会自动覆盖原知识内容。
```

### 17.4 前后对比

验收条件：

```text
AI 整理完成后，用户可以看到原始正文和 AI 整理正文。
桌面端左右对比展示。
移动端上下对比展示。
```

### 17.5 分区应用

验收条件：

```text
用户可以单独应用标题。
用户可以单独应用摘要。
用户可以单独应用正文。
用户可以单独应用标签。
用户可以单独应用分类。
用户可以一键应用全部。
用户取消后，原知识内容不变。
推荐标签超过上限时不会全部静默应用。
推荐分类不存在时不会自动静默创建分类。
```
