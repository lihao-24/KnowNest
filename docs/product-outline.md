# 个人知识库 App 产品总纲 v0.1

## 1. 项目概述

本项目是一个面向个人长期使用的、多端同步的、AI 增强型知识库 App。

它用于统一沉淀用户在生活和工作中的笔记、资料、Prompt、项目经验、链接、复盘、计划和日常记录，并在后续版本中通过 AI 能力提供总结、整理、问答和知识复用能力。

项目第一阶段的目标不是做一个复杂的 Notion 替代品，也不是直接做一个完全依赖 AI 的聊天工具，而是先建立一个稳定、可靠、可长期演进的个人知识系统。

---

## 2. 项目背景

用户日常会在多个场景中产生大量零散信息，例如：

- 工作中的技术笔记、Bug 处理记录、项目经验、开发流程、代码片段
- AI 工具使用经验、Prompt 模板、Claude Code / Codex / ChatGPT 工作流
- 生活中的健康计划、饮食记录、运动计划、旅行计划、学习资料
- 网页文章、文档链接、工具配置说明、临时想法
- 与 AI 对话过程中产生的高价值内容

这些信息如果分散在备忘录、聊天记录、浏览器收藏夹、文档、项目目录和各种 App 中，会带来几个问题：

1. 记录入口分散，后续难以统一检索。
2. 很多有价值的信息无法长期沉淀。
3. 旧经验难以复用，重复思考和重复整理成本高。
4. 生活与工作资料边界不清晰，但又都需要长期保存。
5. AI 对话中产生的内容如果不整理，很快就会丢失上下文价值。

因此，需要一个个人知识库 App，作为用户长期的个人知识沉淀中心。

## 3. 产品定位

### 3.1 一句话定位

一个面向个人长期使用的、多端同步的、AI 增强型知识库 App，用于统一管理生活和工作中的笔记、资料、Prompt、项目经验、链接、复盘和计划。

### 3.2 产品形态

产品初期以 Web App 为主，优先支持浏览器访问和响应式布局，后续逐步扩展为 PWA 和 iOS App。

目标设备包括：

- Mac mini
- MacBook
- iPhone
- 其他可访问浏览器的设备

### 3.3 产品不是

本产品第一阶段不以替代以下产品为目标：

- 不做完整 Notion 替代品
- 不做复杂团队协作工具
- 不做纯聊天式 AI 助手
- 不做大型文档管理系统
- 不做高度复杂的知识图谱工具
- 不做一开始就功能非常重的全平台原生 App

### 3.4 产品更像

本产品更接近于：

- 个人长期记忆库
- 生活与工作资料中心
- AI 编程和日常工作流沉淀系统
- 可检索、可整理、可复用的个人知识操作系统

## 5. 核心目标

### 5.1 第一阶段目标

建立一个稳定、可同步、可持续使用的个人知识记录系统。

第一阶段重点解决：

1. 信息可以快速记录。
2. 信息可以在多个设备间同步。
3. 信息可以按照生活和工作进行粗粒度区分。
4. 信息可以通过标签和搜索进行整理与查找。
5. 信息可以长期保存，后续具备扩展 AI 能力的基础。

### 5.2 长期目标

将产品逐步演进为一个 AI 增强型个人知识操作系统。

长期能力包括：

- 快速记录
- 多设备同步
- 结构化整理
- 全文搜索
- 标签管理
- AI 总结
- AI 标签建议
- AI 问答
- 知识复用
- 周报 / 月报 / 项目复盘生成
- 生活计划和工作流程辅助

## 6. 产品设计原则

### 6.1 长期可用优先于短期炫技

本项目是长期个人项目，不应为了短期技术炫技而牺牲稳定性、可维护性和数据安全。

### 6.2 快速记录优先于复杂整理

知识库的第一价值是让信息进入系统。

如果记录成本过高，用户很难长期坚持。因此第一阶段必须优先保证快速创建、快速保存、快速检索。

### 6.3 多设备同步优先于本地单机体验

用户明确需要在 Mac mini、MacBook、iPhone 和其他设备上使用，因此多设备同步是核心需求。

### 6.4 简单信息架构优先于复杂分类体系

第一阶段只提供少量稳定结构，例如生活 / 工作空间、状态、类型和标签。

避免一开始设计复杂的多级目录、知识图谱和过度分类系统。

### 6.5 Markdown 优先于复杂块编辑器

第一阶段采用 Markdown 作为主要编辑方式，降低开发复杂度，提高内容可迁移性。

后续如果确有必要，再考虑引入块编辑器或富文本能力。

### 6.6 AI 作为增强，不作为第一版核心依赖

AI 是产品长期核心能力，但第一阶段不应依赖复杂 AI 能力才能使用。

第一阶段先把知识记录、同步、整理、搜索做好，后续再逐步增强 AI 能力。

### 6.7 数据安全和可迁移优先

知识库数据具有长期价值，必须重视：

- 数据归属
- 数据备份
- 数据导出
- 数据可迁移
- 数据结构清晰

## 7. 核心使用场景

### 7.1 工作知识沉淀

用户在开发过程中记录技术经验，例如：

- React / Vue 开发经验
- Bug 排查过程
- 组件设计思路
- 性能优化记录
- 项目复盘
- 常用命令
- 工具配置

目标是让工作经验可以被搜索、复用和总结。

### 7.2 AI 工作流沉淀

用户在使用 AI 编程工具时记录：

- Claude Code 使用经验
- Codex 使用经验
- GitHub Copilot 使用经验
- 常用 Prompt
- AI 开发流程
- 模型选择策略
- 代码审查流程

目标是形成自己的 AI 编程方法论。

### 7.3 生活计划管理

用户可以记录生活相关内容，例如：

- 减肥计划
- 饮食记录
- 运动计划
- 学习计划
- 旅行计划
- 日常决策

目标是让生活信息也可以沉淀、回顾和优化。

### 7.4 链接与资料收集

用户可以保存网页、文章、文档、工具链接和参考资料。

第一阶段可以先保存链接和备注，不必实现完整网页抓取。

### 7.5 AI 对话内容归档

用户可以将 ChatGPT / Claude / Codex 对话中的高价值内容复制到知识库中。

例如：

- 技术方案
- Prompt 模板
- 项目规划
- 工作流设计
- 生活计划

目标是避免 AI 对话内容散落在不同聊天记录中。

## 8. 信息架构

### 8.1 顶层结构

第一阶段采用简单、稳定的信息结构：

```text
空间 Space：生活 / 工作
状态 Status：收集箱 / 已整理 / 归档
类型 Type：笔记 / 链接 / Prompt / 项目记录 / 日志 / 摘录 / 计划 / 代码片段
标签 Tags：用户自定义
```

### 8.2 空间 Space

空间用于区分知识的大方向。

初始空间包括：

- 生活 Life
- 工作 Work

后续可以考虑支持自定义空间，但第一阶段不一定需要。

### 8.3 状态 Status

状态用于表示内容的整理阶段。

初始状态包括：

- 收集箱 Inbox：刚记录但尚未整理的内容
- 已整理 Organized：已经补充标题、标签、空间或类型的内容
- 归档 Archived：不常用但需要保留的内容

### 8.4 类型 Type

类型用于标识内容形态。

建议初始类型包括：

- 笔记 Note
- 链接 Link
- Prompt
- 项目记录 Project
- 日志 Log
- 摘录 Excerpt
- 计划 Plan
- 代码片段 Snippet

### 8.5 标签 Tags

标签用于灵活分类。

标签由用户自定义，不限制数量。

示例标签：

```text
React
Vue
Codex
Claude Code
ChatGPT
AI 开发
Bug 修复
性能优化
减肥
饮食
运动
日语
旅行
```

### 8.6 为什么不优先做复杂文件夹

文件夹适合强层级内容，但个人知识往往是多维度的。

例如一条内容可能同时属于：

- 工作
- AI 开发
- Codex
- React
- Prompt

如果使用文件夹，很难同时表达多个维度。因此第一阶段优先采用空间 + 类型 + 标签的组合。

## 9. 核心功能模块

### 9.1 用户认证与同步

支持用户登录，并将数据同步到云端。

第一阶段需要保证：

- 用户可以登录
- 用户只能访问自己的数据
- 多设备登录后可以看到相同内容
- 数据创建、编辑、删除后可以同步

### 9.2 收集箱 Inbox

收集箱是快速记录入口。

所有尚未整理的内容可以先进入收集箱。

核心能力：

- 快速新建内容
- 快速粘贴文本
- 快速保存链接
- 快速保存 Prompt
- 后续再补充空间、类型和标签

设计原则：

- 记录时尽量少填字段
- 整理时再补充结构信息

### 9.3 知识条目管理

支持知识条目的创建、查看、编辑、删除。

基础字段包括：

- 标题
- 正文
- 空间
- 类型
- 状态
- 标签
- 来源链接
- 是否收藏
- 创建时间
- 更新时间

### 9.4 Markdown 编辑器

第一阶段使用 Markdown 编辑方式。

需要支持：

- Markdown 输入
- Markdown 预览
- 代码块
- 标题
- 列表
- 引用
- 链接
- 复制内容

### 9.5 标签系统

支持用户创建和使用标签。

第一阶段能力：

- 给知识条目添加标签
- 删除条目上的标签
- 通过标签筛选内容
- 展示常用标签或最近使用标签

### 9.6 搜索与筛选

第一阶段先做基础全文搜索。

支持：

- 标题搜索
- 正文搜索
- 标签筛选
- 空间筛选
- 类型筛选
- 状态筛选
- 收藏筛选
- 最近更新排序

### 9.7 收藏与归档

收藏用于标记高频使用内容。

归档用于保留低频内容但减少干扰。

### 9.8 AI 辅助整理

AI 能力从后续阶段逐步加入。

优先级较高的 AI 功能包括：

- 生成标题
- 总结当前笔记
- 提炼行动项
- 建议标签
- 整理为结构化笔记

### 9.9 AI 知识库问答

在基础知识库稳定后，加入基于知识库内容的 AI 问答。

目标能力：

- 根据全库内容回答问题
- 根据生活 / 工作空间回答问题
- 根据标签范围回答问题
- 回答时引用相关知识条目
- 支持继续追问

## 10. 多端策略

### 10.1 第一阶段：Web App

优先开发 Web App。

目标：

- Mac mini 可用
- MacBook 可用
- iPhone Safari 可用
- 其他设备浏览器可用

### 10.2 第二阶段：响应式优化与 PWA

在 Web App 基础上增强移动端体验。

目标：

- iPhone 浏览体验良好
- 可添加到主屏幕
- 提供接近 App 的启动体验
- 基础离线缓存能力

### 10.3 第三阶段：iOS App 包装

在 Web 版本稳定后，考虑使用 Capacitor 等方案包装为 iOS App。

目标：

- 更接近原生 App 使用体验
- 支持系统级分享入口
- 支持推送或提醒能力
- 支持更自然的移动端输入体验

### 10.4 暂不优先做的端

第一阶段暂不优先开发：

- 原生 macOS App
- 原生 iOS App
- Android App
- 浏览器插件

这些可以在 Web App 稳定后再评估。

## 11. 技术路线

### 11.1 推荐技术路线

当前推荐路线：

```text
前端框架：Next.js
UI：Tailwind CSS / shadcn/ui
数据库：Supabase PostgreSQL
认证：Supabase Auth
部署：Vercel
编辑器：Markdown 编辑器
AI：OpenAI API 或兼容 API
向量检索：Supabase pgvector，后续版本加入
```

### 11.2 为什么推荐 Next.js

Next.js 适合长期项目，原因包括：

- 适合 Web App
- 后续可扩展 API Routes / Server Actions
- 适合接入 AI 能力
- 适合部署到 Vercel
- 前后端边界较清晰
- 生态成熟

### 11.3 为什么推荐 Supabase

Supabase 适合本项目的原因：

- 提供 PostgreSQL 数据库
- 提供认证能力
- 支持 Row Level Security
- 后续可支持 pgvector
- 适合快速开发个人项目
- 可逐步扩展

### 11.4 为什么第一阶段不选纯本地方案

用户明确需要多设备同步。

如果第一阶段采用 IndexedDB / SQLite 等纯本地方案，后续同步会成为较大改造点。

因此第一阶段直接采用云端数据库更符合长期目标。

## 15. 关键取舍

### 15.1 Web 优先，而不是 App 优先

原因：

- 更快开发
- 更容易多端访问
- 更适合先验证核心需求
- 后续仍可包装成 App

### 15.2 云同步优先，而不是纯本地优先

原因：

- 用户明确需要多设备使用
- 后续 AI 检索更方便
- 避免后续重构同步架构

### 15.3 Markdown 优先，而不是块编辑器优先

原因：

- 开发成本更低
- 数据更可迁移
- 对技术笔记和 Prompt 更友好

### 15.4 标签优先，而不是复杂文件夹优先

原因：

- 个人知识通常是多维度的
- 标签更灵活
- 空间 + 类型 + 标签足以覆盖第一阶段需求

### 15.5 AI 后置，而不是一开始做复杂 RAG

原因：

- 第一阶段应该先保证数据沉淀
- 没有足够数据时，复杂 AI 问答价值有限
- RAG 会显著增加开发复杂度

## 16. 关键风险

### 16.1 需求膨胀风险

知识库产品很容易扩展出大量功能。

应对策略：

- 每个版本只解决一个核心目标
- 严格区分总纲、阶段需求和未来想法
- 第一阶段只做最必要的闭环

### 16.2 使用成本过高风险

如果记录和整理流程太复杂，用户很难坚持使用。

应对策略：

- 优先建设收集箱
- 新建内容时减少必填字段
- 后续用 AI 辅助整理

### 16.3 数据结构频繁变动风险

长期项目如果数据结构不稳定，后续维护成本会增加。

应对策略：

- 第一阶段数据模型保持简单
- 预留扩展字段
- 避免过早抽象复杂关系

### 16.4 AI 成本和稳定性风险

AI 功能可能带来调用成本、响应速度和隐私问题。

应对策略：

- AI 功能从辅助整理开始
- 用户可手动触发 AI，而不是默认自动调用
- 保留非 AI 使用路径

### 16.5 多端体验不一致风险

Web、PWA、iOS App 的体验可能不一致。

应对策略：

- 第一阶段先统一 Web App 体验
- 移动端先做响应式适配
- App 化放到后续阶段

## 18. 当前结论

本项目应采用长期迭代方式推进。

当前最合理的启动方式是：

1. 先完成产品总纲。
2. 基于总纲编写 V0.1 MVP 需求文档。
3. 基于 V0.1 需求文档设计页面和数据表。
4. 使用 Next.js + Supabase + Vercel 实现第一版 Web App。
5. 在基础知识库稳定后，逐步加入 AI 辅助整理和 AI 问答能力。

第一阶段的核心判断是：

```text
先做一个稳定、可同步、可长期使用的 Web 个人知识库；
再逐步把它增强为 AI 驱动的个人知识操作系统。
```
