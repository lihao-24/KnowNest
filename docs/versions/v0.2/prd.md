# KnowNest V0.2 需求文档

## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 产品名称 | KnowNest |
| 版本 | V0.2 |
| 当前状态 | 待开发 |
| 前置版本 | V0.1 已完成并部署到 Vercel |
| 版本定位 | 知识管理体验增强版 |
| 核心目标 | 让 KnowNest 从“基础可用”升级为“更好整理、更好查找、更好维护”的个人知识库产品 |

---

## 2. V0.2 版本背景

KnowNest V0.1 已经完成基础开发，并成功部署到 Vercel。V0.1 的重点是验证产品基础能力，包括用户登录、知识条目管理、基础页面和数据库连接。

V0.2 的重点不是大规模增加复杂能力，而是围绕“知识管理体验”进行增强，让用户可以更方便地整理、查找和维护自己的知识内容。

V0.2 应该优先解决以下问题：

1. 知识条目越来越多后，用户需要搜索能力。
2. 用户需要通过标签、分类来整理知识。
3. 列表页需要更清晰的筛选、排序和展示方式。
4. 需要进一步检查用户数据隔离和 Supabase RLS 权限。
5. 页面体验需要从“能用”优化到“更顺手”。

---

## 3. V0.2 核心目标

### 3.1 产品目标

V0.2 的核心目标是：

> 提升 KnowNest 的知识整理和检索体验，让用户可以更高效地管理个人知识内容。

### 3.2 用户目标

用户在 V0.2 中应该能够：

1. 给知识条目添加标签。
2. 按标签筛选知识。
3. 给知识条目设置分类。
4. 按标题、正文、标签搜索知识。
5. 更方便地查看最近更新的知识。
6. 在手机和电脑上都有较好的使用体验。
7. 确认自己的数据不会被其他用户访问。

### 3.3 技术目标

V0.2 在技术上应完成：

1. 完善知识条目的查询、筛选和排序逻辑。
2. 增加标签和分类相关数据结构。
3. 检查并完善 Supabase RLS 策略。
4. 优化前端状态处理，包括加载、错误、空状态。
5. 保持 Vercel 部署稳定。
6. 保持项目结构清晰，方便后续 V0.3 接入 AI 能力。

## 4. V0.2 功能范围

### 4.1 本版本必做功能

#### 功能一：搜索功能

用户可以在知识列表页搜索已有知识。

##### 需求说明

用户可以通过搜索框输入关键词，系统根据关键词匹配知识条目的：

1. 标题
2. 正文内容
3. 标签名称

##### 交互要求

1. 知识列表页顶部展示搜索框。
2. 用户输入关键词后，列表自动刷新或点击搜索按钮后刷新。
3. 搜索结果为空时，展示空状态提示。
4. 用户可以清空搜索条件，恢复完整列表。
5. 搜索需要和标签筛选、分类筛选兼容。

##### 验收标准

| 编号 | 验收项 |
|---|---|
| S-01 | 输入标题关键词，可以搜索到对应知识 |
| S-02 | 输入正文关键词，可以搜索到对应知识 |
| S-03 | 输入标签关键词，可以搜索到对应知识 |
| S-04 | 搜索结果为空时，有明确空状态提示 |
| S-05 | 清空搜索后，列表恢复正常 |
| S-06 | 搜索结果只展示当前登录用户自己的数据 |

---

#### 功能二：标签系统

用户可以为知识条目添加多个标签，并通过标签筛选知识。

##### 需求说明

每条知识可以拥有多个标签。标签用于帮助用户快速归类和筛选知识。

##### 标签规则

1. 一个知识条目可以有多个标签。
2. 标签名称不能为空。
3. 标签名称不建议超过 20 个字符。
4. 同一用户下不应重复创建完全相同的标签。
5. 不同用户之间的标签数据互不影响。

##### 页面表现

在知识新增和编辑页面中，用户可以：

1. 输入标签名称。
2. 添加多个标签。
3. 删除已添加标签。
4. 保存知识时同步保存标签关系。

在知识列表页中，用户可以：

1. 看到每条知识对应的标签。
2. 点击某个标签进行筛选。
3. 在筛选区域选择一个或多个标签。

##### 验收标准

| 编号 | 验收项 |
|---|---|
| T-01 | 新建知识时可以添加标签 |
| T-02 | 编辑知识时可以修改标签 |
| T-03 | 知识列表中可以展示标签 |
| T-04 | 点击标签可以筛选知识 |
| T-05 | 标签为空时不影响知识保存 |
| T-06 | 同一用户下重复标签不会重复创建 |
| T-07 | 用户只能看到自己的标签 |

---

#### 功能三：分类系统

用户可以为知识条目设置分类，用于更高层级的内容整理。

##### 需求说明

分类用于对知识进行一级归类，例如：

1. 工作
2. 学习
3. 生活
4. 项目
5. 灵感
6. 其他

V0.2 可以先支持默认分类，也可以支持用户自定义分类。建议 V0.2 采用“默认分类 + 自定义分类”的轻量方案。

##### 分类规则

1. 每条知识最多属于一个分类。
2. 分类可以为空。
3. 用户可以创建自定义分类。
4. 用户可以按分类筛选知识。
5. 分类名称不建议超过 20 个字符。

##### 验收标准

| 编号 | 验收项 |
|---|---|
| C-01 | 新建知识时可以选择分类 |
| C-02 | 编辑知识时可以修改分类 |
| C-03 | 知识列表中可以展示分类 |
| C-04 | 用户可以按分类筛选知识 |
| C-05 | 未分类知识可以正常展示 |
| C-06 | 用户只能看到自己的分类 |

---

#### 功能四：知识列表优化

V0.2 需要优化知识列表页，使其更适合管理大量知识内容。

##### 需求说明

知识列表页需要支持：

1. 搜索
2. 标签筛选
3. 分类筛选
4. 排序
5. 空状态
6. 加载状态
7. 移动端适配

##### 排序方式

至少支持以下排序：

1. 最近更新优先
2. 最近创建优先
3. 最早创建优先

默认排序建议为：

```txt
最近更新优先
```

##### 列表卡片展示字段

每个知识卡片建议展示：

1. 标题
2. 内容摘要
3. 分类
4. 标签
5. 创建时间
6. 更新时间
7. 编辑按钮
8. 删除按钮或更多操作按钮

##### 验收标准

| 编号 | 验收项 |
|---|---|
| L-01 | 列表页可以展示知识标题和摘要 |
| L-02 | 列表页可以展示标签 |
| L-03 | 列表页可以展示分类 |
| L-04 | 列表页支持最近更新排序 |
| L-05 | 列表页支持创建时间排序 |
| L-06 | 加载数据时有 loading 状态 |
| L-07 | 没有知识时有空状态提示 |
| L-08 | 手机端列表展示不溢出、不混乱 |

---

#### 功能五：知识详情页优化

V0.2 需要优化知识详情页，使知识内容阅读体验更好。

##### 需求说明

知识详情页应展示：

1. 标题
2. 正文
3. 分类
4. 标签
5. 创建时间
6. 更新时间

如果正文使用 Markdown，详情页应尽量提供基础 Markdown 渲染能力。

V0.2 可以先支持基础 Markdown：

1. 标题
2. 加粗
3. 列表
4. 链接
5. 代码块
6. 引用

##### 验收标准

| 编号 | 验收项 |
|---|---|
| D-01 | 详情页可以正常展示知识内容 |
| D-02 | 标签和分类可以正常展示 |
| D-03 | 可以从详情页进入编辑页 |
| D-04 | 可以从详情页返回列表页 |
| D-05 | Markdown 内容展示基本正常 |
| D-06 | 非当前用户的数据不能被访问 |

#### 功能六：数据权限与 RLS 检查

V0.2 必须重点检查 Supabase 数据权限，确保用户数据隔离。

##### 需求说明

每个用户只能访问、创建、更新、删除自己的数据。

需要检查的数据包括：

1. 知识条目
2. 标签
3. 分类
4. 知识与标签的关联关系

##### 权限要求

| 操作 | 权限要求 |
|---|---|
| 查询知识 | 只能查询自己的知识 |
| 新增知识 | 只能给自己新增知识 |
| 编辑知识 | 只能编辑自己的知识 |
| 删除知识 | 只能删除自己的知识 |
| 查询标签 | 只能查询自己的标签 |
| 查询分类 | 只能查询自己的分类 |

##### 验收标准

| 编号 | 验收项 |
|---|---|
| R-01 | 未登录用户不能访问知识数据 |
| R-02 | 用户 A 不能查询用户 B 的知识 |
| R-03 | 用户 A 不能编辑用户 B 的知识 |
| R-04 | 用户 A 不能删除用户 B 的知识 |
| R-05 | 用户 A 不能访问用户 B 的标签 |
| R-06 | 用户 A 不能访问用户 B 的分类 |
| R-07 | Supabase RLS 已启用并测试通过 |

## 8. 数据结构建议

以下数据结构以常见 Supabase 设计为参考，实际开发时可以根据 V0.1 已有表结构微调。

### 8.1 知识表：`knowledge_items`

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | 主键 |
| user_id | uuid | 所属用户 |
| title | text | 标题 |
| content | text | 正文内容 |
| category_id | uuid / nullable | 分类 ID |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

---

### 8.2 标签表：`tags`

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | 主键 |
| user_id | uuid | 所属用户 |
| name | text | 标签名称 |
| created_at | timestamp | 创建时间 |

建议约束：

```sql
unique(user_id, name)
```

用于避免同一用户重复创建同名标签。

---

### 8.3 知识标签关联表：`knowledge_item_tags`

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | 主键 |
| knowledge_item_id | uuid | 知识 ID |
| tag_id | uuid | 标签 ID |
| created_at | timestamp | 创建时间 |

建议约束：

```sql
unique(knowledge_item_id, tag_id)
```

---

### 8.4 分类表：`categories`

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | 主键 |
| user_id | uuid | 所属用户 |
| name | text | 分类名称 |
| created_at | timestamp | 创建时间 |

建议约束：

```sql
unique(user_id, name)
```

## 9. 搜索与筛选逻辑

### 9.1 搜索条件

搜索关键词需要匹配：

1. `knowledge_items.title`
2. `knowledge_items.content`
3. `tags.name`

V0.2 可以先使用 Supabase 的 `ilike` 实现基础搜索。

示例逻辑：

```txt
title 包含 keyword
或 content 包含 keyword
或 tag name 包含 keyword
```

后续 V0.3 或 V0.4 再考虑更复杂的全文搜索。

### 9.2 筛选条件

知识列表页支持以下筛选：

1. 按分类筛选
2. 按标签筛选
3. 按关键词搜索
4. 按排序方式排序

筛选条件之间的关系建议为：

```txt
关键词搜索 + 分类筛选 + 标签筛选 + 排序
```

也就是说，多个条件同时存在时，展示同时满足条件的知识。

## 10. UI 与交互要求

### 10.1 列表页布局

知识列表页建议分为三部分：

```txt
顶部：页面标题 + 新建知识按钮
中部：搜索框 + 分类筛选 + 标签筛选 + 排序选择
下部：知识卡片列表
```

#### 桌面端

桌面端可以采用：

```txt
左侧筛选区 + 右侧知识列表
```

或：

```txt
顶部筛选栏 + 下方知识列表
```

#### 移动端

移动端建议采用：

```txt
顶部搜索框
下方筛选按钮
再下方知识卡片列表
```

筛选项可以折叠，避免页面过长。

### 10.2 空状态

需要设计以下空状态：

| 场景 | 文案建议 |
|---|---|
| 没有任何知识 | 还没有知识，创建第一条知识吧 |
| 搜索无结果 | 没有找到匹配的知识 |
| 标签为空 | 暂无标签 |
| 分类为空 | 暂无分类 |
| 加载失败 | 数据加载失败，请稍后重试 |

### 10.3 操作反馈

以下操作需要有反馈：

| 操作 | 反馈 |
|---|---|
| 新建知识成功 | 创建成功 |
| 编辑知识成功 | 保存成功 |
| 删除知识成功 | 删除成功 |
| 搜索失败 | 搜索失败，请重试 |
| 权限不足 | 无权访问该内容 |

你现在接手 KnowNest 项目开发。V0.1 已经完成并部署到 Vercel。现在需要开发 V0.2：知识管理体验增强版。

请先阅读项目现有代码结构、数据库调用方式、Supabase 配置和页面路由，然后按照 docs/versions/v0.2/prd.md 的需求进行开发。
